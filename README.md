# وثيقة التصميم والمعمارية البرمجية: تطبيق رفيق العبادة الذكي
# Software Architecture Document (SAD) — Worship Companion Applet

تصف هذه الوثيقة البنية التحتية، المعمارية الهندسية، والمحركات الأساسية لتطبيق **رفيق العبادة الذكي**. تم تصميم هذا التطبيق ليكون رفيق عبادة مخصصًا بالكامل وسياقيًا (Context-Aware) بدلاً من التطبيقات التقليدية القائمة على القوائم الجامدة، مع الالتزام الكامل بمبدأ **الأوفلاين أولاً (Offline-First)** وسرعة الاستجابة القصوى.

---

## 🗺️ الفلسفة المعمارية العامة (Core Philosophy)

التطبيق ليس مجرد مكتبة أرشيفية للكتب أو الأدعية؛ بل هو **رفيق يومي نشط ومساعد روحي رقمي**. تتغير الواجهة وعرض المحتوى ديناميكياً لتلائم:
* **السياق الفعلي للمستخدم**: الطور الزمني الديني (قبل/أثناء/بعد الفريضة) والوقت المتبقي للأذان.
* **التقويم الهجري الشمسي والقمري**: المناسبات الكبرى وتأثيرها المباشر على الأذكار والأعمال المقترحة.
* **فصل كامل للمهام والمسؤوليات**: عزل قاعدة البيانات عن واجهات العرض تماماً عبر طبقة مستودع مركزية (Repository Layer).

---

## 🔄 1. دورة حياة التطبيق الكاملة (Application Lifecycle)

منذ لحظة نقر المستخدم على أيقونة التطبيق وحتى ظهور واجهة المستخدم المستقرة، يمر التطبيق بالدورة المعمارية المتتابعة التالية:

```
[ App Start ] 
      │
      ▼
[ Service Worker Activation ] (تحميل الأصول الساكنة الفورية)
      │
      ▼
[ Load Settings ] (استرجاع إعدادات المدينة والمذهب والحساب من IndexedDB)
      │
      ▼
[ Update & Sync Check ] (التحقق من ملف content_versions ومزامنة الجداول)
      │
      ▼
[ Build Context ] (توليد كائن السياق الحالي: الأوقات، ليلة اليوم، المرحلة الحالية)
      │
      ▼
[ Rule Engine Processing ] (تمرير السياق على محرك القواعد لتحديد الأعمال المناسبة)
      │
      ▼
[ Sort & Priority Resolution ] (رسم الخط الزمني النهائي حسب مستويات الأولوية)
      │
      ▼
[ Render UI & Interactive Checklists ] (تحديث الواجهة الحية للمستخدم)
```

## 🗄️ 2. طبقة مستودع البيانات (Repository Layer)

تم إدخال طبقة **`WorshipRepository`** في `/src/db/repository.ts` لفصل الواجهة الرسومية (React Components) تماماً عن طبقة البيانات المتمثلة في `Dexie (IndexedDB)`. 

### البنية الهيكلية لعزل البيانات:
```
┌─────────────────────────────────────────┐
│         UI Components (React)           │
└────────────────────┬────────────────────┘
                     │ استدعاء واجهات برمجية نظيفة
                     ▼
┌─────────────────────────────────────────┐
│       WorshipRepository Layer           │
└────────────────────┬────────────────────┘
                     │ استعلامات معقدة وعلاقات المحتوى
                     ▼
┌─────────────────────────────────────────┐
│     Dexie.js / IndexedDB (Local Storage)│
└─────────────────────────────────────────┘
```

تتميز هذه الطبقة بـ:
1. **المرونة المستقبلية**: إمكانية استبدال قاعدة البيانات المحلية أو إدخال خادم سحابي دون تعديل سطر واحد في كود الواجهة.
2. **تجميع الاستعلامات وعلاقات المحتوى (Content Relations)**.
3. **توحيد المخرجات البرمجية** لتقليل الأخطاء وثبات الأنواع (Type-Safety).

## 🧠 3. محرك القواعد المطور (Rule Engine & Smart Feed)

تم استبدال نظام الشروط البسيطة بمحرك القواعد المعقد (`RuleEngine` في `/src/utils/contextEngine.ts`) الذي يستطيع مطابقة كتل متكاملة من المدخلات والمحددات.

### مستويات الأولوية الصارمة (Priority Levels Hierarchy):
لمنع تضارب المحتوى وتزاحم المعروض (مثال: توافق ليلة الجمعة مع أول الشهر الهجري ومع فريضة الفجر)، يعتمد الخط الزمني على نظام نقاط دقيق يتراوح كالتالي:

| مستوى الأولوية | القيمة الرقمية | الوصف ومجال الاستخدام |
| :--- | :--- | :--- |
| **CRITICAL** | `100` | المناسبات الاستثنائية والخطيرة (مثل عاشوراء، ليلة القدر، يوم عرفة). |
| **MAJOR_EVENT** | `80` | الأيام والأعياد والمناسبات الهجرية الشهرية. |
| **PRAYER_TIME** | `60` | نوافذ الأذان والصلاة النشطة حالياً. |
| **TAQIBA_WINDOW** | `45` | مرحلة ما بعد الفريضة مباشرة (التعقيبات والأذكار المخصصة). |
| **DAILY_DEVOTIONAL**| `25` | أدعية الأيام وزيارة المعصومين اليومية المحددة. |
| **ALWAYS_ROUTINE** | `10` | الأحاديث الروتينية، الحكم الدورية والدرر اليومية الدوارة. |

### كيفية تقييم الشرط المركب:
يتم التحقق من مطابقة قاعدة العرض عبر محددات ديناميكية مثل `phase` و`isIslamicNight` و`hijriMonth` بشكل فوري عند كل تحديث للوقت، لفرز الأعمال المناسبة وترتيبها تنازلياً وفق الأولوية العليا.

## 📡 4. محرك تحديث ومزامنة البيانات (Sync & Update Engine)

تتم عملية الاستيراد والمزامنة بمبدأ الحفاظ على باقة البيانات واستقرار التطبيق في غياب الإنترنت تماماً.

```
                  فتح التطبيق
                       │
                       ▼
             هل يتوفر إنترنت حالياً؟
             /                  \
          (نعم)                (لا)
           /                      \
تحميل content_versions.json     تحميل قاعدة البيانات المحلية مباشرة
          │                               │
          ▼                               ▼
مقارنة أرقام الإصدارات مع             بدء تشغيل الواجهة الفورية
النسخة المحفوظة في user_settings
          │
          ├───────── لا يوجد تحديث ─────────► تشغيل فوري
          │
          ▼ (تحديث متاح)
تحميل حزم الجداول المتغيرة فقط (مثل hadiths.json)
          │
          ▼
مسح الجداول المستهدفة وتحديثها محلياً (bulkPut)
          │
          ▼
تحديث أرقام الإصدارات محلياً وتشغيل الواجهة
```

## 🔍 5. محرك البحث الذكي المطور (Smart Search Engine)

تم استبدال نظام المطابقة المباشرة والبدائية بمحرك بحث متقدم يقع في `/src/utils/searchEngine.ts` يقوم بحساب أوزان المطابقة والترتيب حسب الصلة والأهمية:

* **مطابقة الاسم/الراوي (الوزن الأقصى: +15)**: إذا كان نص البحث يطابق قائل الحديث أو عنوان الدعاء فإنه يقفز للصدارة.
* **مطابقة الأوسمة والوسوم (الوزن العالي: +10)**: تصنيف الأوسمة يعطي دلالة كبرى على نية المستخدم.
* **مطابقة متن النص (الوزن الأساسي: +5)**: مع احتساب نقاط إضافية (+2) لكل تكرار إضافي للكلمة المفتاحية في جسم النص.
* **تجميع مخرجات الجداول وتصنيفها** (الأحاديث، الحكم، الأدعية والزيارات، المناسبات) في مصفوفات مرتبة ومقيمة برقم إحصائي دقيق.

---

## 📿 6. شبكة علاقات المحتوى (Content Relations)

يحتوي كل عنصر في قاعدة البيانات على شبكة من الأوسمة والوسوم المتطابقة (`tags`) التي يعتمد عليها نظام التوصية وعلاقات البيانات لإظهار:
* أحاديث مرتبطة بفضائل هذا الدعاء المعين.
* حكم ومواعظ تتقاطع مع نفس التصنيف الأخلاقي أو العقائدي للأذكار المعروضة حالياً.
* استعلامات سريعة من خلال `WorshipRepository.getRelatedContent(tags)` توفر للمستخدم عمقاً معرفياً متكاملاً أثناء القراءة.

---

## 🧩 7. تقسيم وتوزيع الخدمات (Modular Services)

بدلاً من دمج جميع المنطق البرمجي في ملف واحد، تم توزيع المسؤوليات على خدمات برمجية مخصصة ومستقلة تماماً:

1. **خدمة مواقيت الصلاة (Prayer Service - `/src/utils/prayerTimes.ts`)**: الحساب الفلكي والزوايا وتحديد اتجاه القبلة والعد التنازلي.
2. **التقويم الهجري الشمسي والقمري (Hijri Service - `/src/utils/hijriCalendar.ts`)**: توليد مصفوفة الأيام الهجرية التناظرية مع الميلادية بدقة متناهية.
3. **مستودع البيانات (WorshipRepository - `/src/db/repository.ts`)**: بوابة الاتصال بالبيانات وعلاقات المحتوى.
4. **محرك البحث المتطور (Search Service - `/src/utils/searchEngine.ts`)**: المطابقة والترتيب والأوزان الذكية.
5. **محرك المزامنة التلقائي (Import & Sync Service - `/src/db/importer.ts`)**: سحب حزم التحديثات والتحقق الآمن.

## 💾 8. إستراتيجية ذاكرة الكاش للوسائط والمستندات (Media Cache Policy)

* يتم تحميل وتخزين جميع النصوص والبيانات محلياً ومباشرة داخل قاعدة بيانات IndexedDB الصلبة.
* **الصور والأيقونات المتجهة**: تستخدم أيقونات الخط المتجه (`lucide-react`) بالكامل لتفادي طلبات الخادم الخارجي والحفاظ على حجم التطبيق والتحميل بدون شبكة.
* **الوضع الخالي من الملقم (Zero Server Overhead)**: لا يقوم التطبيق بأي استدعاءات خارجية لصور عشوائية في أوقات العبادة، لضمان السكينة الرقمية للمستخدم وتوفير طاقة البطارية وباقة البيانات أثناء الاستخدام في المساجد أو السفر.
